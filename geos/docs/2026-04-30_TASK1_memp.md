# Task 1: MemP procedural memory (overnight session)

*2026-04-30 — overnight Task 1 complete. Companion to

## TL;DR

Tested MemP (Fang et al. 2025) — per-trajectory procedural memory with
cosine-similarity retrieval at test time — against the existing M1-u
Dynamic-Cheatsheet-style memory (Suzgun et al. 2025) on the DSv4 17-task
v2 test set. **All memory variants are statistically tied with no-memory
baselines on this task.**

| cell | description | mean | σ | n |
|---|---|---:|---:|:-:|
| C2 | parse-only SR, no RAG, no memory | 0.913 | 0.015 | 3 |
| C5 | C2 + M1-u memory (DC-style, monolithic) | 0.912 | 0.003 | 3 |
| **cMPa** | **C2 + MemP per-task (top-3 retrieval)** | **0.921** | 0.008 | 3 |
| C6 | xmllint hook, no RAG, no memory | 0.921 | 0.006 | 3 |
| C7 | C6 + xmllint MCP tool, no RAG | 0.914 | 0.008 | 3 |
| C11 | C7 + M1-u memory | 0.920 | 0.009 | 3 |
| **cMPb** | **C7 + MemP per-task** | **0.916** | 0.006 | 3 |

## What MemP is (vs Dynamic Cheatsheet)

| | Dynamic Cheatsheet (M1-u) | MemP |
|---|---|---|
| Granularity | one monolithic primer | one entry per train trajectory |
| Distillation | aggregated across 18 train tasks | per-task workflow paragraph |
| Retrieval | none (always inject) | cosine-sim on task descriptions; top-k |
| At test time | constant 820-token primer | top-3 retrievals × ~300-500 tok each |
| Anti-pattern surfacing | aggregated rules | per-task procedural narratives |

For our setup:
- **Build**: each of 18 train trajectories distilled to a workflow
  paragraph via `gemini-3-flash-preview` using MemP's "direct" build
  prompt. Library at `misc/memory_artifacts/memp_dsv4/library.json`
  (18 entries + qwen3-embedding-8b embeddings of task descriptions).
- **Retrieve**: at test time, compute cosine similarity between current
  test task's instructions and each train task's instructions. Take top-3.
- **Render**: concatenate the 3 retrieved workflows into a per-task
  primer at `plugin/memp_per_task/<task>.md`.
- **Inject**: runner now supports `cheatsheet_path_template` with
  `{task}` substitution. `abl_cMP_a_memp_on_c2` + `abl_cMP_b_memp_on_c7`
  use it.

## Why both memory variants land at ≈0 lift

DSv4-flash already produces XML at ~0.91 mean treesim under C2/C7
without any memory. The remaining gap is:
1. **Multi-variant tutorial structure** (e.g., Sneddon's 7 GT files
   across 3 fracture mechanics). Memory primers describe procedural
   patterns but don't tell the agent "produce 7 files".
2. **Composite constitutive recipes** for poromechanics. Memory
   primers contain plausible-sounding but partly hallucinated
   element names.
3. **Schema-vocabulary edge cases** that xmllint catches anyway.

Memory is most valuable when the agent's prior is weak; on DSv4 the
prior is strong enough that retrieved procedural notes overlap with
what the agent would have done anyway.

This matches the broader Task 0 finding: **most plugin components
are null on DSv4-flash; only the xmllint hook contributes anything,
and it's only +0.008.**

## Caveat: distilled content quality

The 18 distilled workflows pass token / hygiene gates (no GT XML
filenames after stripping) but contain **plausible-sounding GEOS
hallucinations** in some entries — `SolidMechanicsLagrangianSSLE`
(real one is `SolidMechanicsLagrangianFEM`), `BiotLinearPoromechanics`
(not a real solver name), `FractureManager`. The "ungrounded" M1-u
distillation had the same issue and was the canonical example of the
risk.

This is a known limitation of self-judged distillation without TreeSim
grounding. A "grounded" MemP variant would require failure-mode signal
in the train trajectories — but our 18 train tasks were all
"success" (mean treesim 0.895 under C2 setup). No failures to ground
on.

## Decision

**Recommend dropping memory entirely from the production stack on DSv4.**
Both M1-u (DC-style) and MemP (per-task) are null over no-memory; both
add wall-time and cost overhead.

If we must include memory for any reason (e.g., paper completeness,
the user's "good number of memory systems tested" goal):
- **MemP per-task slightly preferred over M1-u** for the no-xmllint
  case (+0.009pp at noise level + tighter variance).
- They're indistinguishable in the xmllint case.
- The **mechanism** of MemP is more interesting from a research
  perspective: per-task retrieval scales naturally with library size.
