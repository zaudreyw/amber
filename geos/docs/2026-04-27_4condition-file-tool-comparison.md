## Reading guide / interpretation notes

These notes pre-empt three common questions about the table below.

### Why are RST reads near zero in plugin variants?

The GEOS primer is already in the system prompt — see `run/AGENTS.md` and
`/home/brianliu/.../GEOS_PRIMER.md`. It carries most of the conceptual
material the rsts contain (block structure, common solvers, units, naming
conventions). What the agent actually needs from references is *concrete
syntax and values to imitate*: tag spellings, attribute names, numeric
formats. That lives in XML files, not rst. Even vanilla CC reads ~5×
more XML examples than rsts (8.1 vs 2.1). With RAG, the rst-reading path
collapses entirely and the agent skips straight to (RAG → XML) without
ever opening a doc file.

This has a corollary: the chromadb-coverage gap (87 non-sphinx rst files
not indexed; see `2026-04-27_xmllint-validation-summary.md`) penalizes
plugin variants on tasks whose authoritative references are outside
`src/docs/sphinx/`. Vanilla CC discovers them via Glob/Grep; plugin
variants don't.

### Why does "Tool calls/task" exceed the sum of the file-category columns?

The two measure different things:

- **Tool calls/task** is the total over all tool types: `Read + Bash +
  Write + Glob + Grep + TodoWrite + Edit + WebSearch + Agent + ...`.
- **The file-category columns** (RST sphinx, XML examples, xmllint, …)
  are subsets of *what was read or which Bash command ran*.

For vanilla CC the 37 calls/task break down as: Read 12.4, Bash 11.0,
Write 4.5, Glob 3.1, Grep 2.9, TodoWrite 1.3, WebSearch 1.1, Agent 0.5,
Edit 0.3. The file-category columns only describe the *Read* and the
*Bash-as-file-read* shares; they don't account for Write (output XMLs +
drafts), TodoWrite, WebSearch, or Bash-for-shell-things (mkdir, head, etc.).

#### What does the agent actually read? (vanilla CC, per-task means)

Across 51 task-runs, the 12.4 Read calls/task split into:

| Category | Per task | Note |
|---|---:|---|
| `xml_input_files` | 8.1 | `/geos_lib/inputFiles/...` — examples to imitate |
| `xml_workspace`   | 2.06 | Re-reads of the agent's own drafts in `/workspace/inputs/` |
| `rst_sphinx`      | 1.55 | `/geos_lib/src/docs/sphinx/...` doc files |
| `xsd_schema`      | 0.76 | `geosx_schema.xsd` (the ground-truth grammar) |
| `python`          | 0.75 | e.g. `mandelFigure.py` and other example helpers |
| `other`           | 0.78 | json, txt, geos table files, etc. |
| `rst_nonsphinx`   | 0.55 | The doc files outside `src/docs/sphinx/` (only vanilla finds these) |

The "missing" reads in your question (~2/task on top of XML+RST) are
**`xml_workspace` (re-reading own drafts) + `xsd_schema` reads**. The
schema reads are notable: vanilla CC reads the actual `geosx_schema.xsd`
~0.76 times per task on average without us prompting it to. This is the
agent self-discovering ground-truth grammar.

#### What does the agent actually do with Bash? (vanilla CC, 573 calls)

Categorised by leading command across 3 vanilla-CC minimax m2.7 runs (51
task-runs, 11 Bash calls/task average):

| Category | Count | % | Sample |
|---|---:|---:|---|
| `find` / `ls` / `tree` | 339 | 59.2% | `find /geos_lib/src/docs/sphinx -type f -name "*.rst" -o -name "*.md"` |
| `grep` / `rg` / `egrep` | 147 | 25.7% | `grep -l "QuasiStatic" /geos_lib/inputFiles/**/*.xml` |
| `mkdir` | 38 | 6.6% | `mkdir -p /workspace/inputs/scripts` |
| Read-equivalents (`cat`/`head`/`sed`/`awk`) | 29 | 5.1% | `head -60 /workspace/inputs/.../base.xml` |
| `python3` | 6 | 1.0% | `python3 -c "import xml.etree.ElementTree as ET; …"` |
| `xmllint` | 3 | 0.5% | `xmllint --noout /workspace/inputs/...` |
| `wc` / `cp` / `mv` | 9 | 1.6% | various housekeeping |
| Other (echo, pwd, env, …) | 2 | 0.3% | — |

**Key insight: 85% of Bash calls in vanilla CC are filesystem search
(`find`+`ls`+`grep`+`rg`).** That's 9.5 search calls/task via Bash, on
top of the 6.0 search calls/task via the dedicated Glob+Grep tools. So
**the agent does ~15.5 filesystem-search operations per task, more than
its 12.4 Read calls.** Vanilla CC's main activity is *searching*, not
reading. With RAG, both numbers collapse — 0-1 Glob/task and Bash drops
similarly because the agent goes straight from RAG hit → Read.

Note also: only 0.5% of Bash is `xmllint`. The agent rarely uses Bash
for validation. mkdir is exactly the 38 = 51 task-runs × ~0.75/task we'd
expect for "create inputs/outputs/scripts directories" boilerplate.

### Where does the xmllint signal come from?

`xmllint` is in `run/AGENTS.md` (line 551, "Validating Input Files"
section) and the GEOS primer (line 525) — both are inlined into the
system prompt. **It is *not* exposed as a tool**; the agent calls it via
`Bash`. Cross-run, 91 invocations across 87 task-runs (~10% of tasks).
The `InputXMLFiles.rst` doc — which separately documents xmllint usage
— is read in only **2 task-runs out of 908**, and **none of those two
overlapped with an xmllint invocation**. So:

- The xmllint usage signal is driven by the system-prompt mention, not
  by doc discovery.
- Despite being in the prompt, only ~10% of task-runs actually invoke
  it. The mention is buried inside a "validating input files" subsection
  alongside two other validators (`geosx -s`, `geosx -v`); the agent
  often skips it.
- If we want >10% adoption, the right move is either (a) wire it into
  the existing Stop hook so every emitted XML is automatically schema-
  validated, or (b) expose it as a first-class MCP tool. The
  recommendation in `xmllint_validation.md` is "both, with a 4-cell
  ablation."

The schema path itself is not in the prompt — the agent has to
discover that `schema.xsd` lives at
`/geos_lib/src/coreComponents/schema/schema.xsd`. Adding the absolute
path to the prompt would remove a source of friction even before any
hook/tool work.
