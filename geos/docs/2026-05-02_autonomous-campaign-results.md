### Why does RAG hurt?

This reproduces the pattern from the 2026-04-29 ablation: the RAG
MCP tools (search_navigator, search_schema, search_technical) cause
the agent to substitute prose-chunk retrievals for full-file Reads
of `/geos_lib/inputFiles/`. The auto-generated tool-call distribution
table below confirms it:

- **Non-RAG cells (F0, F2, F4, F6)**: ~50% of tool calls are Read,
  ~20% Grep, ~12% Glob ‚Äî heavy filesystem exploration of the
  example library.
- **RAG cells (F1, F3, F5, F7)**: only 26-34% Read, 2-5% Glob/Grep,
  but **15-20% RAG calls combined** (search_navigator +
  search_schema + search_technical). The agent substitutes RAG
  retrievals for Glob/Grep/Read and ends up with less complete
  structural context.

DSv4-flash is good enough at filesystem exploration that the RAG
shortcut is a regression. The 2026-04-29 ablation reached the same
conclusion; this campaign confirms it under the new
`AGENTS.md`/primer split.

### What the result means in plain terms

**Once `AGENTS.md` (post-strip) is reduced to a clean harness
contract, DSv4-flash needs no plugin, no methodology prescription,
and no additional guidance to reach ~0.92 quality on this 17-task
benchmark.** F0 (literally nothing on top of AGENTS.md + 5-line
primer) scores 0.910 ¬± 0.019. F4/F2/SE/F6 all add 1pp through
modest combinations of stop-hook validation, xmllint, memory, or
self-evolved tooling ‚Äî small enough that the choice is dominated by
non-quality criteria (token cost, wall-time efficiency, complexity).

The plugin's value on DSv4 is **not** quality lift. It's:
- Stop-hook backstop for retry-on-empty-output (essential for
  weaker models like gpt-oss-120b; redundant for DSv4)
- xmllint MCP tool for proactive validation (small +pp)
- Memory cheatsheet for vocabulary priming (small +pp)
- Self-evolved plugin (SE) ‚Äî same band, demonstrates that an
  agent-authored plugin transfers without regression

The only knob that matters in the *negative* direction is RAG;
turning it off recovers quality.

## Why no-plugin is so competitive

2. **DSv4 is good enough at first-principles GEOS authoring** that

## Bottleneck analysis (paper input)

A 3-stage pipeline (`scripts/bottleneck/`) classifies every (cell,
seed, task) into a structural failure category by mining
`treesim_detail` and feeding per-task diagnostics to DSv4-flash, then
aggregating with DSv4-pro into a paper-ready narrative. Results across
F0/F2/F4/F6/SE √ó 3 seeds √ó 17 tasks = 255 tasks live in
`docs/XN-019_bottleneck-analysis-phase2.md` and
`/data/matt/bn_phase2/stage3/`.

Key takeaway from DSv4-pro synthesis (full text in XN-019):

- **Baseline (F0) failures** concentrate in `Solvers` (failure weight
  0.70, agent invents solver names like `thermoPoromechanicalSolver`
  and adds extraneous attributes), `Geometry` (0.86, even small
  coordinate errors break structural validity), `Constitutive` (0.67,
  +1 dummy material), and complete `<Events>` block omission (10/51
  baseline tasks). 12 tasks have `bad_attribute_value`, 9 have
  `extra_block`, 6 have `missing_block`.
- **Best-config (F4) impact** is sharply *targeted*: `missing_block`
  drops 6‚Üí3 and `partial_implementation` 7‚Üí6 (block-presence
  validation works); but `extra_block` (9‚Üí11) and
  `hallucinated_extras` (4‚Üí7) get *worse*, and `bad_attribute_value`
  is unchanged (12‚Üí11). Solvers failure weight balloons to 1.58
  because once missing blocks are fixed, residual mistakes
  concentrate in solver-content errors. The fraction of flawless
  tasks does not increase.
- **Adapter design implications**: (i) structural validation must be
  paired with child-count and forbidden-element rules ‚Äî block
  presence alone is not sufficient; (ii) need a solver-type oracle
  cross-referencing attribute names against the GEOS schema (this
  attacks the 1.58 Solvers failure weight directly); (iii) `Geometry`
  and `Outputs` need templated enforcement (forced output types,
  geometry dry-run); (iv) closed-loop retries driven by validator
  output are needed because static hooks alone don't repair
  attribute-level errors.

Pipeline doc: `docs/2026-05-02_bottleneck-analysis-pipeline.md`.

## Where to look next: F0 trajectory study

The user asked: can we study F0 trajectories to find efficiency
wins? Yes. The 18% gap between F0 and SE suggests the F0 agent is
doing redundant exploration that the SE plugin's skills/memory have
already compressed. Three concrete things to look at:

1. **Recurring file-read patterns.** From the file-read aggregate
   in `2026-05-01_C0_abs_min_primer_behavior.md` (a similar prior
   ablation), the agent re-reads the same triaxial-driver,
   wellbore, and poromechanics example XMLs across many tasks. These
   are deterministic mappings (task X ‚Üí most-relevant 2-3 example
   XMLs); they could be a lookup table baked into the primer or a
   memory module.
2. **Recurring Grep patterns.** Top Grep patterns are CamelCase
   identifiers: `TriaxialDriver`, `DruckerPrager`, `TableFunction`,
   `InternalMesh`, `SolidMechanicsLagrangianSSLE`,
   `SinglePhasePoromechanics`. These are stable across seeds and
   tasks. A "GEOS-name ‚Üí file-path" memory would replace ~50% of
   Grep volume.
3. **Verification reads.** The agent reads its own
   `/workspace/inputs/*.xml` back ~1.6 times/run (emergent
   verification ‚Äî never instructed). This is the Stop hook's job;
   when the Stop hook is on, the read-backs become redundant. The
   stop-hook cells (F2, F6) DO show fewer Read calls.

A focused study would:
- pick 2-3 tasks where F0 took 90+ turns
- diff F0's trajectory against SE's on the same task
- identify which F0 turns were redundant (e.g. re-reading the same
  example XML, repeated Grep for the same identifier)
- propose a skill or memory module that absorbs that pattern

**This study has been done** ‚Äî see
`docs/2026-05-02_F0_vs_SE_trajectory_diff.md`. Key findings:
- F0 averages 73 tools-before-first-Write; SE averages 59. The 13-tool
  savings is concentrated in the *exploration phase* (file discovery).
- F0 averages 2.3 read-back verifications per task; SE averages 0.2.
  The Stop hook absorbs verification (a 90% reduction in this metric
  alone ‚Äî invisible in the quality score because S effect was ‚àí0.003).
- Top redundant reads: schema.xsd (11√ó), constitutive .hpp headers
  (~6√ó each), canonical example XMLs (~5-9√ó). v3's cheatsheet
  compresses these.
- Three cross-pair patterns extracted (DeepSeek summary):
  cheatsheet-driven file discovery, task-specific skill shortcuts,
  parallel targeted Agent calls.

Concretely:
- "task-family lookup" memory: 17 tasks ‚Üí the 2-3 most-relevant
  `/geos_lib/inputFiles/*.xml` for each
- "constitutive-name lookup" memory: ~30 GEOS class names ‚Üí
  `<header.hpp>:line` references
- Could potentially shave 30-50% off F0's turn budget without
  touching quality.

### Per-task standouts (qwen)

- **TutorialPoroelasticity**: A=0.346 ‚Üí B=**0.689** (**+0.343**). On
  the qwen-hardest task, augmentations recovered roughly half the gap

  base capability is limited."

  perfect.
- **ExampleEDPWellbore**: A=0.954 ‚Üí B=0.998 (+0.044). Augmentations
  finished a near-correct draft.

### Failure mode (Phase B, ExampleIsothermalLeakyWell)

Hit 1800s timeout at 121 tool calls, never wrote the canonical XMLs.
Trajectory shows the agent spent its budget on auxiliary support files
(PVT data, coordinate/pressure `.geos` data, Python data-generation
scripts, visualization scripts) before writing a single XML deck. The
xmllint MCP and plugin appear to encourage the agent to over-prepare.
This is consistent with B's higher tool count (~110 vs A's ~75) and
suggests a **wall-time risk** introduced by the augmentation stack ‚Äî
worth surfacing in the paper's discussion of trade-offs.

### Implications for the paper

1. **Regime-dependence claim is supported**: qwen baseline is 3pp lower
   than DSv4 baseline as expected; augmentation effect on qwen
   (+0.008 common, +0.343 on the hardest task) is at least as large as
   on DSv4 (+0.011), with much of the lift concentrated on tasks where
   the base model struggles.
2. **Wall-time trade-off**: the augmented stack exhibits a non-trivial
   timeout risk (1/17 = 6% of tasks on qwen) that doesn't appear on
   DSv4. Acknowledge as a cost of augmentation in the limitations.
3. **N=1 seed caveat**: with one seed, the per-task Œî values are noisy.
   The +0.343 TutorialPoroelasticity lift is the paper's eye-catching
   anecdote; the +0.008 mean is the conservative claim. We ran only 1
   seed because that is what the paper plan ¬ß5 calls for and to stay
   within budget.
