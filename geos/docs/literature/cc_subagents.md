Create custom subagents

Copy page

Create and use specialized AI subagents in Claude Code for task-specific workflows and improved context management.

Copy page

> Documentation Index
> -------------------
>
> Fetch the complete documentation index at: <https://code.claude.com/docs/llms.txt>
>
> Use this file to discover all available pages before exploring further.

<span data-as="p">Subagents are specialized AI assistants that handle specific types of tasks. Use one when a side task would flood your main conversation with search results, logs, or file contents you won’t reference again: the subagent does that work in its own context and returns only the summary. Define a custom subagent when you keep spawning the same kind of worker with the same instructions.</span> <span data-as="p">Each subagent runs in its own context window with a custom system prompt, specific tool access, and independent permissions. When Claude encounters a task that matches a subagent’s description, it delegates to that subagent, which works independently and returns results. To see the context savings in practice, the <a href="/docs/en/context-window" class="link">context window visualization</a> walks through a session where a subagent handles research in its own separate window.</span>

<span data-as="p">If you need multiple agents working in parallel and communicating with each other, see <a href="/docs/en/agent-teams" class="link">agent teams</a> instead. Subagents work within a single session; agent teams coordinate across separate sessions.</span>

<span data-as="p">Subagents help you:</span>

-   **Preserve context** by keeping exploration and implementation out of your main conversation
-   **Enforce constraints** by limiting which tools a subagent can use
-   **Reuse configurations** across projects with user-level subagents
-   **Specialize behavior** with focused system prompts for specific domains
-   **Control costs** by routing tasks to faster, cheaper models like Haiku

<span data-as="p">Claude uses each subagent’s description to decide when to delegate tasks. When you create a subagent, write a clear description so Claude knows when to use it.</span> <span data-as="p">Claude Code includes several built-in subagents like **Explore**, **Plan**, and **general-purpose**. You can also create custom subagents to handle specific tasks. This page covers:</span>

<span class="cursor-pointer">Built-in subagents</span>

<span data-as="p">Claude Code includes built-in subagents that Claude automatically uses when appropriate. Each inherits the parent conversation’s permissions with additional tool restrictions.</span>

-   Explore

-   Plan

-   General-purpose

-   Other

<span data-as="p">A fast, read-only agent optimized for searching and analyzing codebases.</span>

-   **Model**: Haiku (fast, low-latency)
-   **Tools**: Read-only tools (denied access to Write and Edit tools)
-   **Purpose**: File discovery, code search, codebase exploration

<span data-as="p">Claude delegates to Explore when it needs to search or understand a codebase without making changes. This keeps exploration results out of your main conversation context.</span><span data-as="p">When invoking Explore, Claude specifies a thoroughness level: **quick** for targeted lookups, **medium** for balanced exploration, or **very thorough** for comprehensive analysis.</span>

<span data-as="p">A research agent used during <a href="/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis" class="link">plan mode</a> to gather context before presenting a plan.</span>

-   **Model**: Inherits from main conversation
-   **Tools**: Read-only tools (denied access to Write and Edit tools)
-   **Purpose**: Codebase research for planning

<span data-as="p">When you’re in plan mode and Claude needs to understand your codebase, it delegates research to the Plan subagent. This prevents infinite nesting (subagents cannot spawn other subagents) while still gathering necessary context.</span>

<span data-as="p">A capable agent for complex, multi-step tasks that require both exploration and action.</span>

-   **Model**: Inherits from main conversation
-   **Tools**: All tools
-   **Purpose**: Complex research, multi-step operations, code modifications

<span data-as="p">Claude delegates to general-purpose when the task requires both exploration and modification, complex reasoning to interpret results, or multiple dependent steps.</span>

<span data-as="p">Claude Code includes additional helper agents for specific tasks. These are typically invoked automatically, so you don’t need to use them directly.</span>

### 

<a href="#use-the-/agents-command" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Use the /agents command</span>

<span data-as="p">The `/agents` command opens a tabbed interface for managing subagents. The **Running** tab shows live subagents and lets you open or stop them. The **Library** tab lets you:</span>

-   View all available subagents (built-in, user, project, and plugin)
-   Create new subagents with guided setup or Claude generation
-   Edit existing subagent configuration and tool access
-   Delete custom subagents
-   See which subagents are active when duplicates exist

<span data-as="p">This is the recommended way to create and manage subagents. For manual creation or automation, you can also add subagent files directly.</span> <span data-as="p">To list all configured subagents from the command line without starting an interactive session, run `claude agents`. This shows agents grouped by source and indicates which are overridden by higher-priority definitions.</span>

### 

<a href="#choose-the-subagent-scope" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Choose the subagent scope</span>

<span data-as="p">Subagents are Markdown files with YAML frontmatter. Store them in different locations depending on scope. When multiple subagents share the same name, the higher-priority location wins.</span>

<table><thead><tr class="header"><th style="text-align: left;">Location</th><th style="text-align: left;">Scope</th><th style="text-align: left;">Priority</th><th style="text-align: left;">How to create</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;">Managed settings</td><td style="text-align: left;">Organization-wide</td><td style="text-align: left;">1 (highest)</td><td style="text-align: left;">Deployed via <a href="/docs/en/settings" class="link">managed settings</a></td></tr><tr class="even"><td style="text-align: left;"><code>--agents</code> CLI flag</td><td style="text-align: left;">Current session</td><td style="text-align: left;">2</td><td style="text-align: left;">Pass JSON when launching Claude Code</td></tr><tr class="odd"><td style="text-align: left;"><code>.claude/agents/</code></td><td style="text-align: left;">Current project</td><td style="text-align: left;">3</td><td style="text-align: left;">Interactive or manual</td></tr><tr class="even"><td style="text-align: left;"><code>~/.claude/agents/</code></td><td style="text-align: left;">All your projects</td><td style="text-align: left;">4</td><td style="text-align: left;">Interactive or manual</td></tr><tr class="odd"><td style="text-align: left;">Plugin’s <code>agents/</code> directory</td><td style="text-align: left;">Where plugin is enabled</td><td style="text-align: left;">5 (lowest)</td><td style="text-align: left;">Installed with <a href="/docs/en/plugins" class="link">plugins</a></td></tr></tbody></table>

<span data-as="p">**Project subagents** (`.claude/agents/`) are ideal for subagents specific to a codebase. Check them into version control so your team can use and improve them collaboratively.</span> <span data-as="p">Project subagents are discovered by walking up from the current working directory. Directories added with `--add-dir` <a href="/docs/en/permissions#additional-directories-grant-file-access-not-configuration" class="link">grant file access only</a> and are not scanned for subagents. To share subagents across projects, use `~/.claude/agents/` or a <a href="/docs/en/plugins" class="link">plugin</a>.</span> <span data-as="p">**User subagents** (`~/.claude/agents/`) are personal subagents available in all your projects.</span> <span data-as="p">**CLI-defined subagents** are passed as JSON when launching Claude Code. They exist only for that session and aren’t saved to disk, making them useful for quick testing or automation scripts. You can define multiple subagents in a single `--agents` call:</span>

    claude --agents '{
      "code-reviewer": {
        "description": "Expert code reviewer. Use proactively after code changes.",
        "prompt": "You are a senior code reviewer. Focus on code quality, security, and best practices.",
        "tools": ["Read", "Grep", "Glob", "Bash"],
        "model": "sonnet"
      },
      "debugger": {
        "description": "Debugging specialist for errors and test failures.",
        "prompt": "You are an expert debugger. Analyze errors, identify root causes, and provide fixes."
      }
    }'

<span data-as="p">The `--agents` flag accepts JSON with the same <a href="#supported-frontmatter-fields" class="link">frontmatter</a> fields as file-based subagents: `description`, `prompt`, `tools`, `disallowedTools`, `model`, `permissionMode`, `mcpServers`, `hooks`, `maxTurns`, `skills`, `initialPrompt`, `memory`, `effort`, `background`, `isolation`, and `color`. Use `prompt` for the system prompt, equivalent to the markdown body in file-based subagents.</span> <span data-as="p">**Managed subagents** are deployed by organization administrators. Place markdown files in `.claude/agents/` inside the <a href="/docs/en/settings#settings-files" class="link">managed settings directory</a>, using the same frontmatter format as project and user subagents. Managed definitions take precedence over project and user subagents with the same name.</span> <span data-as="p">**Plugin subagents** come from <a href="/docs/en/plugins" class="link">plugins</a> you’ve installed. They appear in `/agents` alongside your custom subagents. See the <a href="/docs/en/plugins-reference#agents" class="link">plugin components reference</a> for details on creating plugin subagents.</span>

<span data-as="p">For security reasons, plugin subagents do not support the `hooks`, `mcpServers`, or `permissionMode` frontmatter fields. These fields are ignored when loading agents from a plugin. If you need them, copy the agent file into `.claude/agents/` or `~/.claude/agents/`. You can also add rules to <a href="/docs/en/settings#permission-settings" class="link"><code>permissions.allow</code></a> in `settings.json` or `settings.local.json`, but these rules apply to the entire session, not just the plugin subagent.</span>

<span data-as="p">Subagent definitions from any of these scopes are also available to <a href="/docs/en/agent-teams#use-subagent-definitions-for-teammates" class="link">agent teams</a>: when spawning a teammate, you can reference a subagent type and the teammate uses its `tools` and `model`, with the definition’s body appended to the teammate’s system prompt as additional instructions. See <a href="/docs/en/agent-teams#use-subagent-definitions-for-teammates" class="link">agent teams</a> for which frontmatter fields apply on that path.</span>

### 

<a href="#write-subagent-files" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Write subagent files</span>

<span data-as="p">Subagent files use YAML frontmatter for configuration, followed by the system prompt in Markdown:</span>

<span data-as="p">Subagents are loaded at session start. If you create a subagent by manually adding a file, restart your session or use `/agents` to load it immediately.</span>

    ---
    name: code-reviewer
    description: Reviews code for quality and best practices
    tools: Read, Glob, Grep
    model: sonnet
    ---

    You are a code reviewer. When invoked, analyze the code and provide
    specific, actionable feedback on quality, security, and best practices.

<span data-as="p">The frontmatter defines the subagent’s metadata and configuration. The body becomes the system prompt that guides the subagent’s behavior. Subagents receive only this system prompt (plus basic environment details like working directory), not the full Claude Code system prompt.</span> <span data-as="p">A subagent starts in the main conversation’s current working directory. Within a subagent, `cd` commands do not persist between Bash or PowerShell tool calls and do not affect the main conversation’s working directory. To give the subagent an isolated copy of the repository instead, set <a href="#supported-frontmatter-fields" class="link"><code>isolation: worktree</code></a>.</span>

### 

<a href="#choose-a-model" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Choose a model</span>

<span data-as="p">The `model` field controls which <a href="/docs/en/model-config" class="link">AI model</a> the subagent uses:</span>

-   **Model alias**: Use one of the available aliases: `sonnet`, `opus`, or `haiku`
-   **Full model ID**: Use a full model ID such as `claude-opus-4-7` or `claude-sonnet-4-6`. Accepts the same values as the `--model` flag
-   **inherit**: Use the same model as the main conversation
-   **Omitted**: If not specified, defaults to `inherit` (uses the same model as the main conversation)

<span data-as="p">When Claude invokes a subagent, it can also pass a `model` parameter for that specific invocation. Claude Code resolves the subagent’s model in this order:</span>

1.  The <a href="/docs/en/model-config#environment-variables" class="link"><code>CLAUDE_CODE_SUBAGENT_MODEL</code></a> environment variable, if set
2.  The per-invocation `model` parameter
3.  The subagent definition’s `model` frontmatter
4.  The main conversation’s model

### 

<a href="#control-subagent-capabilities" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Control subagent capabilities</span>

<span data-as="p">You can control what subagents can do through tool access, permission modes, and conditional rules.</span>

#### 

<a href="#available-tools" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Available tools</span>

<span data-as="p">Subagents can use any of Claude Code’s <a href="/docs/en/tools-reference" class="link">internal tools</a>. By default, subagents inherit all tools from the main conversation, including MCP tools.</span> <span data-as="p">To restrict tools, use either the `tools` field (allowlist) or the `disallowedTools` field (denylist). This example uses `tools` to exclusively allow Read, Grep, Glob, and Bash. The subagent can’t edit files, write files, or use any MCP tools:</span>

    ---
    name: safe-researcher
    description: Research agent with restricted capabilities
    tools: Read, Grep, Glob, Bash
    ---

<span data-as="p">This example uses `disallowedTools` to inherit every tool from the main conversation except Write and Edit. The subagent keeps Bash, MCP tools, and everything else:</span>

    ---
    name: no-writes
    description: Inherits every tool except file writes
    disallowedTools: Write, Edit
    ---

<span data-as="p">If both are set, `disallowedTools` is applied first, then `tools` is resolved against the remaining pool. A tool listed in both is removed.</span>

#### 

<a href="#restrict-which-subagents-can-be-spawned" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Restrict which subagents can be spawned</span>

<span data-as="p">When an agent runs as the main thread with `claude --agent`, it can spawn subagents using the Agent tool. To restrict which subagent types it can spawn, use `Agent(agent_type)` syntax in the `tools` field.</span>

In version 2.1.63, the Task tool was renamed to Agent. Existing `Task(...)` references in settings and agent definitions still work as aliases.

    ---
    name: coordinator
    description: Coordinates work across specialized agents
    tools: Agent(worker, researcher), Read, Bash
    ---

<span data-as="p">This is an allowlist: only the `worker` and `researcher` subagents can be spawned. If the agent tries to spawn any other type, the request fails and the agent sees only the allowed types in its prompt. To block specific agents while allowing all others, use <a href="#disable-specific-subagents" class="link"><code>permissions.deny</code></a> instead.</span> <span data-as="p">To allow spawning any subagent without restrictions, use `Agent` without parentheses:</span>

    tools: Agent, Read, Bash

<span data-as="p">If `Agent` is omitted from the `tools` list entirely, the agent cannot spawn any subagents. This restriction only applies to agents running as the main thread with `claude --agent`. Subagents cannot spawn other subagents, so `Agent(agent_type)` has no effect in subagent definitions.</span>

#### 

<a href="#scope-mcp-servers-to-a-subagent" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Scope MCP servers to a subagent</span>

<span data-as="p">Use the `mcpServers` field to give a subagent access to <a href="/docs/en/mcp" class="link">MCP</a> servers that aren’t available in the main conversation. Inline servers defined here are connected when the subagent starts and disconnected when it finishes. String references share the parent session’s connection.</span>

<span data-as="p">The `mcpServers` field applies in both contexts where an agent file can run:</span>

-   As a subagent, spawned through the Agent tool or an @-mention
-   As the main session, launched with <a href="#invoke-subagents-explicitly" class="link"><code>--agent</code></a> or the `agent` setting

<span data-as="p">When the agent is the main session, inline server definitions connect at startup alongside servers from <a href="/docs/en/mcp" class="link"><code>.mcp.json</code></a> and settings files.</span>

<span data-as="p">Each entry in the list is either an inline server definition or a string referencing an MCP server already configured in your session:</span>

    ---
    name: browser-tester
    description: Tests features in a real browser using Playwright
    mcpServers:
      # Inline definition: scoped to this subagent only
      - playwright:
          type: stdio
          command: npx
          args: ["-y", "@playwright/mcp@latest"]
      # Reference by name: reuses an already-configured server
      - github
    ---

    Use the Playwright tools to navigate, screenshot, and interact with pages.

<span data-as="p">Inline definitions use the same schema as `.mcp.json` server entries (`stdio`, `http`, `sse`, `ws`), keyed by the server name.</span> <span data-as="p">To keep an MCP server out of the main conversation entirely and avoid its tool descriptions consuming context there, define it inline here rather than in `.mcp.json`. The subagent gets the tools; the parent conversation does not.</span>

#### 

<a href="#permission-modes" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Permission modes</span>

<span data-as="p">The `permissionMode` field controls how the subagent handles permission prompts. Subagents inherit the permission context from the main conversation and can override the mode, except when the parent mode takes precedence as described below.</span>

<table><thead><tr class="header"><th style="text-align: left;">Mode</th><th style="text-align: left;">Behavior</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;"><code>default</code></td><td style="text-align: left;">Standard permission checking with prompts</td></tr><tr class="even"><td style="text-align: left;"><code>acceptEdits</code></td><td style="text-align: left;">Auto-accept file edits and common filesystem commands for paths in the working directory or <code>additionalDirectories</code></td></tr><tr class="odd"><td style="text-align: left;"><code>auto</code></td><td style="text-align: left;"><a href="/docs/en/permission-modes#eliminate-prompts-with-auto-mode" class="link">Auto mode</a>: a background classifier reviews commands and protected-directory writes</td></tr><tr class="even"><td style="text-align: left;"><code>dontAsk</code></td><td style="text-align: left;">Auto-deny permission prompts (explicitly allowed tools still work)</td></tr><tr class="odd"><td style="text-align: left;"><code>bypassPermissions</code></td><td style="text-align: left;">Skip permission prompts</td></tr><tr class="even"><td style="text-align: left;"><code>plan</code></td><td style="text-align: left;">Plan mode (read-only exploration)</td></tr></tbody></table>

<span data-as="p">Use `bypassPermissions` with caution. It skips permission prompts, allowing the subagent to execute operations without approval. Writes to `.git`, `.claude`, `.vscode`, `.idea`, and `.husky` directories still prompt for confirmation, except for `.claude/commands`, `.claude/agents`, and `.claude/skills`. See <a href="/docs/en/permission-modes#skip-all-checks-with-bypasspermissions-mode" class="link">permission modes</a> for details.</span>

<span data-as="p">If the parent uses `bypassPermissions` or `acceptEdits`, this takes precedence and cannot be overridden. If the parent uses <a href="/docs/en/permission-modes#eliminate-prompts-with-auto-mode" class="link">auto mode</a>, the subagent inherits auto mode and any `permissionMode` in its frontmatter is ignored: the classifier evaluates the subagent’s tool calls with the same block and allow rules as the parent session.</span>

#### 

<a href="#preload-skills-into-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Preload skills into subagents</span>

<span data-as="p">Use the `skills` field to inject skill content into a subagent’s context at startup. This gives the subagent domain knowledge without requiring it to discover and load skills during execution.</span>

    ---
    name: api-developer
    description: Implement API endpoints following team conventions
    skills:
      - api-conventions
      - error-handling-patterns
    ---

    Implement API endpoints. Follow the conventions and patterns from the preloaded skills.

<span data-as="p">The full content of each skill is injected into the subagent’s context, not just made available for invocation. Subagents don’t inherit skills from the parent conversation; you must list them explicitly.</span> <span data-as="p">You cannot preload skills that set <a href="/docs/en/skills#control-who-invokes-a-skill" class="link"><code>disable-model-invocation: true</code></a>, since preloading draws from the same set of skills Claude can invoke. If a listed skill is missing or disabled, Claude Code skips it and logs a warning to the debug log.</span>

<span data-as="p">This is the inverse of <a href="/docs/en/skills#run-skills-in-a-subagent" class="link">running a skill in a subagent</a>. With `skills` in a subagent, the subagent controls the system prompt and loads skill content. With `context: fork` in a skill, the skill content is injected into the agent you specify. Both use the same underlying system.</span>

#### 

<a href="#enable-persistent-memory" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Enable persistent memory</span>

<span data-as="p">The `memory` field gives the subagent a persistent directory that survives across conversations. The subagent uses this directory to build up knowledge over time, such as codebase patterns, debugging insights, and architectural decisions.</span>

    ---
    name: code-reviewer
    description: Reviews code for quality and best practices
    memory: user
    ---

    You are a code reviewer. As you review code, update your agent memory with
    patterns, conventions, and recurring issues you discover.

<span data-as="p">Choose a scope based on how broadly the memory should apply:</span>

<table><thead><tr class="header"><th style="text-align: left;">Scope</th><th style="text-align: left;">Location</th><th style="text-align: left;">Use when</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;"><code>user</code></td><td style="text-align: left;"><code>~/.claude/agent-memory/&lt;name-of-agent&gt;/</code></td><td style="text-align: left;">the subagent should remember learnings across all projects</td></tr><tr class="even"><td style="text-align: left;"><code>project</code></td><td style="text-align: left;"><code>.claude/agent-memory/&lt;name-of-agent&gt;/</code></td><td style="text-align: left;">the subagent’s knowledge is project-specific and shareable via version control</td></tr><tr class="odd"><td style="text-align: left;"><code>local</code></td><td style="text-align: left;"><code>.claude/agent-memory-local/&lt;name-of-agent&gt;/</code></td><td style="text-align: left;">the subagent’s knowledge is project-specific but should not be checked into version control</td></tr></tbody></table>

<span data-as="p">When memory is enabled:</span>

-   The subagent’s system prompt includes instructions for reading and writing to the memory directory.
-   The subagent’s system prompt also includes the first 200 lines or 25KB of `MEMORY.md` in the memory directory, whichever comes first, with instructions to curate `MEMORY.md` if it exceeds that limit.
-   Read, Write, and Edit tools are automatically enabled so the subagent can manage its memory files.

##### Persistent memory tips

-   <span data-as="p">`project` is the recommended default scope. It makes subagent knowledge shareable via version control. Use `user` when the subagent’s knowledge is broadly applicable across projects, or `local` when the knowledge should not be checked into version control.</span>
-   <span data-as="p">Ask the subagent to consult its memory before starting work: “Review this PR, and check your memory for patterns you’ve seen before.”</span>
-   <span data-as="p">Ask the subagent to update its memory after completing a task: “Now that you’re done, save what you learned to your memory.” Over time, this builds a knowledge base that makes the subagent more effective.</span>
-   <span data-as="p">Include memory instructions directly in the subagent’s markdown file so it proactively maintains its own knowledge base:</span>

        Update your agent memory as you discover codepaths, patterns, library
        locations, and key architectural decisions. This builds up institutional
        knowledge across conversations. Write concise notes about what you found
        and where.

#### 

<a href="#conditional-rules-with-hooks" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Conditional rules with hooks</span>

<span data-as="p">For more dynamic control over tool usage, use `PreToolUse` hooks to validate operations before they execute. This is useful when you need to allow some operations of a tool while blocking others.</span> <span data-as="p">This example creates a subagent that only allows read-only database queries. The `PreToolUse` hook runs the script specified in `command` before each Bash command executes:</span>

    ---
    name: db-reader
    description: Execute read-only database queries
    tools: Bash
    hooks:
      PreToolUse:
        - matcher: "Bash"
          hooks:
            - type: command
              command: "./scripts/validate-readonly-query.sh"
    ---

<span data-as="p">Claude Code <a href="/docs/en/hooks#pretooluse-input" class="link">passes hook input as JSON</a> via stdin to hook commands. The validation script reads this JSON, extracts the Bash command, and <a href="/docs/en/hooks#exit-code-2-behavior-per-event" class="link">exits with code 2</a> to block write operations:</span>

    #!/bin/bash
    # ./scripts/validate-readonly-query.sh

    INPUT=$(cat)
    COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

    # Block SQL write operations (case-insensitive)
    if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE)\b' > /dev/null; then
      echo "Blocked: Only SELECT queries are allowed" >&2
      exit 2
    fi

    exit 0

<span data-as="p">See <a href="/docs/en/hooks#pretooluse-input" class="link">Hook input</a> for the complete input schema and <a href="/docs/en/hooks#exit-code-output" class="link">exit codes</a> for how exit codes affect behavior.</span>

#### 

<a href="#disable-specific-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Disable specific subagents</span>

<span data-as="p">You can prevent Claude from using specific subagents by adding them to the `deny` array in your <a href="/docs/en/settings#permission-settings" class="link">settings</a>. Use the format `Agent(subagent-name)` where `subagent-name` matches the subagent’s name field.</span>

    {
      "permissions": {
        "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
      }
    }

<span data-as="p">This works for both built-in and custom subagents. You can also use the `--disallowedTools` CLI flag:</span>

    claude --disallowedTools "Agent(Explore)"

<span data-as="p">See <a href="/docs/en/permissions#tool-specific-permission-rules" class="link">Permissions documentation</a> for more details on permission rules.</span>

### 

<a href="#define-hooks-for-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Define hooks for subagents</span>

<span data-as="p">Subagents can define <a href="/docs/en/hooks" class="link">hooks</a> that run during the subagent’s lifecycle. There are two ways to configure hooks:</span>

1.  **In the subagent’s frontmatter**: Define hooks that run only while that subagent is active
2.  **In `settings.json`**: Define hooks that run in the main session when subagents start or stop

#### 

<a href="#hooks-in-subagent-frontmatter" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Hooks in subagent frontmatter</span>

<span data-as="p">Define hooks directly in the subagent’s markdown file. These hooks only run while that specific subagent is active and are cleaned up when it finishes.</span>

<span data-as="p">Frontmatter hooks fire when the agent is spawned as a subagent through the Agent tool or an @-mention, and when the agent runs as the main session via <a href="#invoke-subagents-explicitly" class="link"><code>--agent</code></a> or the `agent` setting. In the main-session case they run alongside any hooks defined in <a href="/docs/en/hooks" class="link"><code>settings.json</code></a>.</span>

<span data-as="p">All <a href="/docs/en/hooks#hook-events" class="link">hook events</a> are supported. The most common events for subagents are:</span>

<table><thead><tr class="header"><th style="text-align: left;">Event</th><th style="text-align: left;">Matcher input</th><th style="text-align: left;">When it fires</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;"><code>PreToolUse</code></td><td style="text-align: left;">Tool name</td><td style="text-align: left;">Before the subagent uses a tool</td></tr><tr class="even"><td style="text-align: left;"><code>PostToolUse</code></td><td style="text-align: left;">Tool name</td><td style="text-align: left;">After the subagent uses a tool</td></tr><tr class="odd"><td style="text-align: left;"><code>Stop</code></td><td style="text-align: left;">(none)</td><td style="text-align: left;">When the subagent finishes (converted to <code>SubagentStop</code> at runtime)</td></tr></tbody></table>

<span data-as="p">This example validates Bash commands with the `PreToolUse` hook and runs a linter after file edits with `PostToolUse`:</span>

    ---
    name: code-reviewer
    description: Review code changes with automatic linting
    hooks:
      PreToolUse:
        - matcher: "Bash"
          hooks:
            - type: command
              command: "./scripts/validate-command.sh $TOOL_INPUT"
      PostToolUse:
        - matcher: "Edit|Write"
          hooks:
            - type: command
              command: "./scripts/run-linter.sh"
    ---

<span data-as="p">When the agent is invoked as a subagent, `Stop` hooks in frontmatter are automatically converted to `SubagentStop` events.</span>

#### 

<a href="#project-level-hooks-for-subagent-events" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Project-level hooks for subagent events</span>

<span data-as="p">Configure hooks in `settings.json` that respond to subagent lifecycle events in the main session.</span>

<table><thead><tr class="header"><th style="text-align: left;">Event</th><th style="text-align: left;">Matcher input</th><th style="text-align: left;">When it fires</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;"><code>SubagentStart</code></td><td style="text-align: left;">Agent type name</td><td style="text-align: left;">When a subagent begins execution</td></tr><tr class="even"><td style="text-align: left;"><code>SubagentStop</code></td><td style="text-align: left;">Agent type name</td><td style="text-align: left;">When a subagent completes</td></tr></tbody></table>

<span data-as="p">Both events support matchers to target specific agent types by name. This example runs a setup script only when the `db-agent` subagent starts, and a cleanup script when any subagent stops:</span>

    {
      "hooks": {
        "SubagentStart": [
          {
            "matcher": "db-agent",
            "hooks": [
              { "type": "command", "command": "./scripts/setup-db-connection.sh" }
            ]
          }
        ],
        "SubagentStop": [
          {
            "hooks": [
              { "type": "command", "command": "./scripts/cleanup-db-connection.sh" }
            ]
          }
        ]
      }
    }

<span data-as="p">See <a href="/docs/en/hooks" class="link">Hooks</a> for the complete hook configuration format.</span>

<a href="#work-with-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Work with subagents</span>

### 

<a href="#understand-automatic-delegation" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Understand automatic delegation</span>

<span data-as="p">Claude automatically delegates tasks based on the task description in your request, the `description` field in subagent configurations, and current context. To encourage proactive delegation, include phrases like “use proactively” in your subagent’s description field.</span>

### 

<a href="#invoke-subagents-explicitly" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Invoke subagents explicitly</span>

<span data-as="p">When automatic delegation isn’t enough, you can request a subagent yourself. Three patterns escalate from a one-off suggestion to a session-wide default:</span>

-   **Natural language**: name the subagent in your prompt; Claude decides whether to delegate
-   **@-mention**: guarantees the subagent runs for one task
-   **Session-wide**: the whole session uses that subagent’s system prompt, tool restrictions, and model via the `--agent` flag or the `agent` setting

<span data-as="p">For natural language, there’s no special syntax. Name the subagent and Claude typically delegates:</span>

    Use the test-runner subagent to fix failing tests
    Have the code-reviewer subagent look at my recent changes

<span data-as="p">**@-mention the subagent.** Type `@` and pick the subagent from the typeahead, the same way you @-mention files. This ensures that specific subagent runs rather than leaving the choice to Claude:</span>

    @"code-reviewer (agent)" look at the auth changes

<span data-as="p">Your full message still goes to Claude, which writes the subagent’s task prompt based on what you asked. The @-mention controls which subagent Claude invokes, not what prompt it receives.</span> <span data-as="p">Subagents provided by an enabled <a href="/docs/en/plugins" class="link">plugin</a> appear in the typeahead as `<plugin-name>:<agent-name>`. Named background subagents currently running in the session also appear in the typeahead, showing their status next to the name. You can also type the mention manually without using the picker: `@agent-<name>` for local subagents, or `@agent-<plugin-name>:<agent-name>` for plugin subagents.</span> <span data-as="p">**Run the whole session as a subagent.** Pass <a href="/docs/en/cli-reference" class="link"><code>--agent &lt;name&gt;</code></a> to start a session where the main thread itself takes on that subagent’s system prompt, tool restrictions, and model:</span>

    claude --agent code-reviewer

<span data-as="p">The subagent’s system prompt replaces the default Claude Code system prompt entirely, the same way <a href="/docs/en/cli-reference" class="link"><code>--system-prompt</code></a> does. `CLAUDE.md` files and project memory still load through the normal message flow. The agent name appears as `@<name>` in the startup header so you can confirm it’s active.</span> <span data-as="p">This works with built-in and custom subagents, and the choice persists when you resume the session.</span> <span data-as="p">For a plugin-provided subagent, pass the scoped name: `claude --agent <plugin-name>:<agent-name>`.</span> <span data-as="p">To make it the default for every session in a project, set `agent` in `.claude/settings.json`:</span>

    {
      "agent": "code-reviewer"
    }

<span data-as="p">The CLI flag overrides the setting if both are present.</span>

### 

<a href="#run-subagents-in-foreground-or-background" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Run subagents in foreground or background</span>

<span data-as="p">Subagents can run in the foreground (blocking) or background (concurrent):</span>

-   **Foreground subagents** block the main conversation until complete. Permission prompts and clarifying questions (like <a href="/docs/en/tools-reference" class="link"><code>AskUserQuestion</code></a>) are passed through to you.
-   **Background subagents** run concurrently while you continue working. Before launching, Claude Code prompts for any tool permissions the subagent will need, ensuring it has the necessary approvals upfront. Once running, the subagent inherits these permissions and auto-denies anything not pre-approved. If a background subagent needs to ask clarifying questions, that tool call fails but the subagent continues.

<span data-as="p">If a background subagent fails due to missing permissions, you can start a new foreground subagent with the same task to retry with interactive prompts.</span> <span data-as="p">Claude decides whether to run subagents in the foreground or background based on the task. You can also:</span>

-   Ask Claude to “run this in the background”
-   Press **Ctrl+B** to background a running task

<span data-as="p">To disable all background task functionality, set the `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS` environment variable to `1`. See <a href="/docs/en/env-vars" class="link">Environment variables</a>.</span> <span data-as="p">When <a href="#fork-the-current-conversation" class="link">fork mode</a> is enabled, every subagent spawn runs in the background regardless of the `background` field. Forks still surface permission prompts in your terminal as they occur instead of pre-approving; named subagents follow the pre-approval flow above.</span>

### 

<a href="#common-patterns" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Common patterns</span>

#### 

<a href="#isolate-high-volume-operations" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Isolate high-volume operations</span>

<span data-as="p">One of the most effective uses for subagents is isolating operations that produce large amounts of output. Running tests, fetching documentation, or processing log files can consume significant context. By delegating these to a subagent, the verbose output stays in the subagent’s context while only the relevant summary returns to your main conversation.</span>

    Use a subagent to run the test suite and report only the failing tests with their error messages

#### 

<a href="#run-parallel-research" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Run parallel research</span>

<span data-as="p">For independent investigations, spawn multiple subagents to work simultaneously:</span>

    Research the authentication, database, and API modules in parallel using separate subagents

<span data-as="p">Each subagent explores its area independently, then Claude synthesizes the findings. This works best when the research paths don’t depend on each other.</span>

<span data-as="p">When subagents complete, their results return to your main conversation. Running many subagents that each return detailed results can consume significant context.</span>

<span data-as="p">For tasks that need sustained parallelism or exceed your context window, <a href="/docs/en/agent-teams" class="link">agent teams</a> give each worker its own independent context.</span>

#### 

<a href="#chain-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Chain subagents</span>

<span data-as="p">For multi-step workflows, ask Claude to use subagents in sequence. Each subagent completes its task and returns results to Claude, which then passes relevant context to the next subagent.</span>

    Use the code-reviewer subagent to find performance issues, then use the optimizer subagent to fix them

### 

<a href="#choose-between-subagents-and-main-conversation" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Choose between subagents and main conversation</span>

<span data-as="p">Use the **main conversation** when:</span>

-   The task needs frequent back-and-forth or iterative refinement
-   Multiple phases share significant context (planning → implementation → testing)
-   You’re making a quick, targeted change
-   Latency matters. Subagents start fresh and may need time to gather context

<span data-as="p">Use **subagents** when:</span>

-   The task produces verbose output you don’t need in your main context
-   You want to enforce specific tool restrictions or permissions
-   The work is self-contained and can return a summary

<span data-as="p">Consider <a href="/docs/en/skills" class="link">Skills</a> instead when you want reusable prompts or workflows that run in the main conversation context rather than isolated subagent context.</span> <span data-as="p">For a quick question about something already in your conversation, use <a href="/docs/en/interactive-mode#side-questions-with-%2Fbtw" class="link"><code>/btw</code></a> instead of a subagent. It sees your full context but has no tool access, and the answer is discarded rather than added to history.</span>

<span data-as="p">Subagents cannot spawn other subagents. If your workflow requires nested delegation, use <a href="/docs/en/skills" class="link">Skills</a> or <a href="#chain-subagents" class="link">chain subagents</a> from the main conversation.</span>

### 

<a href="#manage-subagent-context" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Manage subagent context</span>

#### 

<a href="#resume-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Resume subagents</span>

<span data-as="p">Each subagent invocation creates a new instance with fresh context. To continue an existing subagent’s work instead of starting over, ask Claude to resume it.</span> <span data-as="p">Resumed subagents retain their full conversation history, including all previous tool calls, results, and reasoning. The subagent picks up exactly where it stopped rather than starting fresh.</span> <span data-as="p">When a subagent completes, Claude receives its agent ID. Claude uses the `SendMessage` tool with the agent’s ID as the `to` field to resume it. The `SendMessage` tool is only available when <a href="/docs/en/agent-teams" class="link">agent teams</a> are enabled via `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`.</span> <span data-as="p">To resume a subagent, ask Claude to continue the previous work:</span>

    Use the code-reviewer subagent to review the authentication module
    [Agent completes]

    Continue that code review and now analyze the authorization logic
    [Claude resumes the subagent with full context from previous conversation]

<span data-as="p">If a stopped subagent receives a `SendMessage`, it auto-resumes in the background without requiring a new `Agent` invocation.</span> <span data-as="p">You can also ask Claude for the agent ID if you want to reference it explicitly, or find IDs in the transcript files at `~/.claude/projects/{project}/{sessionId}/subagents/`. Each transcript is stored as `agent-{agentId}.jsonl`.</span> <span data-as="p">Subagent transcripts persist independently of the main conversation:</span>

-   **Main conversation compaction**: When the main conversation compacts, subagent transcripts are unaffected. They’re stored in separate files.
-   **Session persistence**: Subagent transcripts persist within their session. You can <a href="#resume-subagents" class="link">resume a subagent</a> after restarting Claude Code by resuming the same session.
-   **Automatic cleanup**: Transcripts are cleaned up based on the `cleanupPeriodDays` setting (default: 30 days).

#### 

<a href="#auto-compaction" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Auto-compaction</span>

<span data-as="p">Subagents support automatic compaction using the same logic as the main conversation. By default, auto-compaction triggers at approximately 95% capacity. To trigger compaction earlier, set `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` to a lower percentage (for example, `50`). See <a href="/docs/en/env-vars" class="link">environment variables</a> for details.</span> <span data-as="p">Compaction events are logged in subagent transcript files:</span>

    {
      "type": "system",
      "subtype": "compact_boundary",
      "compactMetadata": {
        "trigger": "auto",
        "preTokens": 167189
      }
    }

<span data-as="p">The `preTokens` value shows how many tokens were used before compaction occurred.</span>

<a href="#fork-the-current-conversation" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Fork the current conversation</span>

<span data-as="p">Forked subagents are experimental and require Claude Code v2.1.117 or later. Behavior and configuration may change in future releases. Enable them by setting the <a href="/docs/en/env-vars" class="link"><code>CLAUDE_CODE_FORK_SUBAGENT</code></a> environment variable to `1`. The variable is honored in interactive mode and via the SDK or `claude -p`.</span>

<span data-as="p">A fork is a subagent that inherits the entire conversation so far instead of starting fresh. This drops the input isolation that subagents otherwise provide: a fork sees the same system prompt, tools, model, and message history as the main session, so you can hand it a side task without re-explaining the situation. The fork’s own tool calls still stay out of your conversation and only its final result comes back, so your main context window stays clean. Use a fork when a named subagent would need too much background to be useful, or when you want to try several approaches in parallel from the same starting point.</span> <span data-as="p">Enabling fork mode changes Claude Code in three ways:</span>

-   Claude spawns a fork whenever it would otherwise use the <a href="#built-in-subagents" class="link">general-purpose</a> subagent. Named subagents such as Explore still spawn as before.
-   Every subagent spawn runs in the <a href="#run-subagents-in-foreground-or-background" class="link">background</a>, whether it is a fork or a named subagent. Set `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS` to `1` to keep spawns synchronous.
-   The `/fork` command spawns a fork instead of acting as an alias for <a href="/docs/en/commands" class="link"><code>/branch</code></a>.

<span data-as="p">You can start a fork yourself with `/fork` followed by a directive. Claude Code names the fork from the first words of the directive. The following example forks the conversation to draft test cases while you continue with the implementation in the main session:</span>

    /fork draft unit tests for the parser changes so far

<span data-as="p">The fork appears in a panel below your prompt and runs in the background while you keep working. When it finishes, its result arrives as a message in your main conversation. The next section covers the panel controls for watching and steering forks while they run.</span>

### 

<a href="#observe-and-steer-running-forks" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Observe and steer running forks</span>

<span data-as="p">Running forks appear in a panel below the prompt input, with one row for the main session and one for each fork. Use these keys to interact with the panel:</span>

<table><thead><tr class="header"><th style="text-align: left;">Key</th><th style="text-align: left;">Action</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;"><code>↑</code> / <code>↓</code></td><td style="text-align: left;">Move between rows</td></tr><tr class="even"><td style="text-align: left;"><code>Enter</code></td><td style="text-align: left;">Open the selected fork’s transcript and send it follow-up messages</td></tr><tr class="odd"><td style="text-align: left;"><code>x</code></td><td style="text-align: left;">Dismiss a finished fork or stop a running one</td></tr><tr class="even"><td style="text-align: left;"><code>Esc</code></td><td style="text-align: left;">Return focus to the prompt input</td></tr></tbody></table>

### 

<a href="#how-forks-differ-from-named-subagents" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">How forks differ from named subagents</span>

<span data-as="p">A fork inherits everything the main session has at the moment it spawns. A named subagent starts from its own definition.</span>

<table><thead><tr class="header"><th style="text-align: left;"></th><th style="text-align: left;">Fork</th><th style="text-align: left;">Named subagent</th></tr></thead><tbody><tr class="odd"><td style="text-align: left;">Context</td><td style="text-align: left;">Full conversation history</td><td style="text-align: left;">Fresh context with the prompt you pass</td></tr><tr class="even"><td style="text-align: left;">System prompt and tools</td><td style="text-align: left;">Same as main session</td><td style="text-align: left;">From the subagent’s <a href="#write-subagent-files" class="link">definition file</a></td></tr><tr class="odd"><td style="text-align: left;">Model</td><td style="text-align: left;">Same as main session</td><td style="text-align: left;">From the subagent’s <code>model</code> field</td></tr><tr class="even"><td style="text-align: left;">Permissions</td><td style="text-align: left;">Prompts surface in your terminal</td><td style="text-align: left;"><a href="#run-subagents-in-foreground-or-background" class="link">Pre-approved</a> before launch, then auto-denied</td></tr><tr class="odd"><td style="text-align: left;">Prompt cache</td><td style="text-align: left;">Shared with main session</td><td style="text-align: left;">Separate cache</td></tr></tbody></table>

<span data-as="p">Because a fork’s system prompt and tool definitions are identical to the parent, its first request reuses the parent’s prompt cache. This makes forking cheaper than spawning a fresh subagent for tasks that need the same context.</span> <span data-as="p">When Claude spawns a fork through the Agent tool, it can pass `isolation: "worktree"` so the fork’s file edits are written to a separate git worktree instead of your checkout.</span>

### 

<a href="#limitations" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Limitations</span>

<span data-as="p">Setting `CLAUDE_CODE_FORK_SUBAGENT=1` enables fork mode in interactive sessions, <a href="/docs/en/headless" class="link">non-interactive mode</a>, and the Agent SDK. A fork cannot spawn further forks.</span>

<span class="cursor-pointer">Example subagents</span>

<span data-as="p">These examples demonstrate effective patterns for building subagents. Use them as starting points, or generate a customized version with Claude.</span>

<span data-as="p">**Best practices:**</span>

<a href="#code-reviewer" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Code reviewer</span>

<span data-as="p">A read-only subagent that reviews code without modifying it. This example shows how to design a focused subagent with limited tool access (no Edit or Write) and a detailed prompt that specifies exactly what to look for and how to format output.</span>

    ---
    name: code-reviewer
    description: Expert code review specialist. Proactively reviews code for quality, security, and maintainability. Use immediately after writing or modifying code.
    tools: Read, Grep, Glob, Bash
    model: inherit
    ---

    You are a senior code reviewer ensuring high standards of code quality and security.

    When invoked:
    1. Run git diff to see recent changes
    2. Focus on modified files
    3. Begin review immediately

    Review checklist:
    - Code is clear and readable
    - Functions and variables are well-named
    - No duplicated code
    - Proper error handling
    - No exposed secrets or API keys
    - Input validation implemented
    - Good test coverage
    - Performance considerations addressed

    Provide feedback organized by priority:
    - Critical issues (must fix)
    - Warnings (should fix)
    - Suggestions (consider improving)

    Include specific examples of how to fix issues.

<span class="cursor-pointer">Debugger</span>

<span data-as="p">A subagent that can both analyze and fix issues. Unlike the code reviewer, this one includes Edit because fixing bugs requires modifying code. The prompt provides a clear workflow from diagnosis to verification.</span>

    ---
    name: debugger
    description: Debugging specialist for errors, test failures, and unexpected behavior. Use proactively when encountering any issues.
    tools: Read, Edit, Bash, Grep, Glob
    ---

    You are an expert debugger specializing in root cause analysis.

    When invoked:
    1. Capture error message and stack trace
    2. Identify reproduction steps
    3. Isolate the failure location
    4. Implement minimal fix
    5. Verify solution works

    Debugging process:
    - Analyze error messages and logs
    - Check recent code changes
    - Form and test hypotheses
    - Add strategic debug logging
    - Inspect variable states

    For each issue, provide:
    - Root cause explanation
    - Evidence supporting the diagnosis
    - Specific code fix
    - Testing approach
    - Prevention recommendations

    Focus on fixing the underlying issue, not the symptoms.

### 

<a href="#database-query-validator" class="-ml-10 flex items-center opacity-0 border-0 group-hover:opacity-100 focus:opacity-100 focus:outline-0 group/link">​</a>

<span class="cursor-pointer">Database query validator</span>

<span data-as="p">A subagent that allows Bash access but validates commands to permit only read-only SQL queries. This example shows how to use `PreToolUse` hooks for conditional validation when you need finer control than the `tools` field provides.</span>

    ---
    name: db-reader
    description: Execute read-only database queries. Use when analyzing data or generating reports.
    tools: Bash
    hooks:
      PreToolUse:
        - matcher: "Bash"
          hooks:
            - type: command
              command: "./scripts/validate-readonly-query.sh"
    ---

    You are a database analyst with read-only access. Execute SELECT queries to answer questions about the data.

    When asked to analyze data:
    1. Identify which tables contain the relevant data
    2. Write efficient SELECT queries with appropriate filters
    3. Present results clearly with context

    You cannot modify data. If asked to INSERT, UPDATE, DELETE, or modify schema, explain that you only have read access.

<span data-as="p">Claude Code <a href="/docs/en/hooks#pretooluse-input" class="link">passes hook input as JSON</a> via stdin to hook commands. The validation script reads this JSON, extracts the command being executed, and checks it against a list of SQL write operations. If a write operation is detected, the script <a href="/docs/en/hooks#exit-code-2-behavior-per-event" class="link">exits with code 2</a> to block execution and returns an error message to Claude via stderr.</span> <span data-as="p">Create the validation script anywhere in your project. The path must match the `command` field in your hook configuration:</span>

    #!/bin/bash
    # Blocks SQL write operations, allows SELECT queries

    # Read JSON input from stdin
    INPUT=$(cat)

    # Extract the command field from tool_input using jq
    COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

    if [ -z "$COMMAND" ]; then
      exit 0
    fi

    # Block write operations (case-insensitive)
    if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE|REPLACE|MERGE)\b' > /dev/null; then
      echo "Blocked: Write operations not allowed. Use SELECT queries only." >&2
      exit 2
    fi

    exit 0

<span data-as="p">Make the script executable:</span>

    chmod +x ./scripts/validate-readonly-query.sh

<span data-as="p">The hook receives JSON via stdin with the Bash command in `tool_input.command`. Exit code 2 blocks the operation and feeds the error message back to Claude. See <a href="/docs/en/hooks#exit-code-output" class="link">Hooks</a> for details on exit codes and <a href="/docs/en/hooks#pretooluse-input" class="link">Hook input</a> for the complete input schema.</span>
