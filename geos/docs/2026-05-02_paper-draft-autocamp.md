# Adapting a coding agent for GEOS XML authoring: when do harness adaptations matter?

*Generated by research-copilot. This is a draft of the methods, results,
and discussion sections; numbers and figures are paper-ready, prose is
not yet polished.*

## Methods

### Benchmark and task

We evaluate coding agents on a GEOS multiphysics XML authoring task.
Each task instance provides a natural-language simulation specification
(rock mechanics, poromechanics, hydraulic fracture, thermal coupling,
etc.) and a list of XML files to author under `/workspace/inputs/`.
Ground truth (GT) is the reference XML set from the GEOS source tree;
the agent has no access to GT during the run. The `/geos_lib/` mount
provides read-only access to the GEOS source repository (~6.4k example
XMLs and source headers); decontamination removes the GT files and
their immediate neighbors from `/geos_lib/`.

**Quality metric.** We report **TreeSim**, a recursive tree-similarity
score (0-1 scale, 1 = exact GT match). For each (gen, GT) XML pair, we
align elements by `(tag, name)`, score attribute matches at each level,
and recurse into children with size-weighted aggregation. We aggregate
per-task TreeSim across the file set produced by an agent
(typically 2-7 files per task), then average across tasks per cell.
The conservative metric is **failures-as-zero mean**: tasks that
produce no parseable XML are scored 0 rather than excluded.

### Three evaluation sets

| set | n_tasks | source | seen during memory distillation? | seen during plugin self-evolution? |
|---|---:|---|:-:|:-:|
| test-17 | 17 | held-out subset of `experiments_test36_template` | no | no |
| ICL-10 | 10 | `experiments_from_mined_specs` (novel specs) | **no** | **no** |
| train-19 | 19 | the rest of `experiments_test36_template` | **yes** (m1u distilled here) | **yes** (v3 evolved here) |

The three sets exercise different distributions:
- **test-17** is the standard internal benchmark; cells were tuned
  against this set.
- **ICL-10** is genuinely out-of-distribution. It contains tasks the
  agent's memory and self-evolved plugin have never been exposed to.
- **train-19** is in-distribution for memory/plugin cells. We restrict
  evaluation to memory-free cells (F0, F6) on this set to avoid the
  contamination.

### Cells (harness configurations)

We evaluate a Resolution-IV 2^(4-1) fractional factorial over four
harness factors plus three additional cells, all on `deepseek-v4-flash`
through Claude Code 2.1.119 with a stripped `AGENTS.md` (1.6 KB
post-strip — environment contract only) and a 5-line GEOS primer:

| cell | RAG | SR-hook | xmllint MCP | memory cheatsheet | other |
|---|:-:|:-:|:-:|:-:|---|
| F0 | – | – | – | – | baseline (no plugin) |
| F1 | + | – | – | + | |
| F2 | – | + | – | + | |
| F3 | + | + | – | – | |
| F4 | – | – | + | + | autocamp winner on test-17 |
| F5 | + | – | + | – | |
| F6 | – | + | + | – | |
| F7 | + | + | + | + | full stack |
| F8 | – | + | + | + | factorial-completion cell (S+X+M, no RAG) |
| F11 | – | + | + | + | F6 + v3 PRIMER + v3 cheatsheet (decomposed SE prose only) |
| SE | – | + | + | + | self-evolved v3 plugin (skills disabled at runtime; PRIMER + cheatsheet + hooks/MCP) |

**Factor definitions**:
- **R** (RAG): three MCP servers — `search_navigator`, `search_schema`,
  `search_technical` — providing retrieval over GEOS docs/schema.
- **S** (SR-hook): a Stop-hook that forces a self-refine pass before
  exit (validates output presence, reads back generated XML).
- **X** (xmllint MCP): an `mcp__xmllint__validate_geos_xml` tool that
  schema-validates against the GEOS XSD.
- **M** (memory cheatsheet): the m1u cheatsheet — physics→solver
  routing rules + anti-patterns — distilled offline from train-18
  trajectories with `gemini-3-flash-preview`.

For SE, a separate "self-evolving" pipeline iteratively rewrites the
plugin contents (PRIMER, skills, hooks) on training trajectories;
**v3** is the version validated against round-0 tasks. At eval time
the runtime disables `Skill` invocation (via `--disallowedTools Skill`)
to isolate the contribution of the prose primer + cheatsheet + hooks.
F11 holds memory and prose constant with SE but uses only the v3 prose
without packaging-level changes.

### Bottleneck-analysis pipeline (DSv4-flash + DSv4-pro)

To attribute aggregate quality to specific structural failure modes,
we built a 3-stage analysis pipeline:

1. **Stage 1 (no LLM)** — `scripts/bottleneck/extract.py` mines the
   recursive `treesim_detail` tree from the scorer, ranking the top-K
   failing subtrees by `(1 - score) × (n_gt_children + 1)` (impact =
   quality loss × subtree size); also extracts trajectory features
   (tool counts, file re-reads, xmllint calls, edit churn,
   grep/glob query distributions) from each `events.jsonl`.

2. **Stage 2 (DSv4-flash)** — `scripts/bottleneck/llm_per_task.py`
   sends the diagnostic plus a focused GT-vs-gen XML excerpt for the
   worst-scoring section (when score < 0.7) to DSv4-flash with a
   strict-JSON schema. Output fields: `failure_category` ∈
   {missing_block, extra_block, hallucinated_extras, structural_mismatch,
   bad_attribute_value, partial_implementation, wrong_constitutive,
   no_failure}, `primary_failure_section`, `root_cause`,
   `trajectory_evidence`, `would_have_helped`, `severity`.

3. **Stage 3 (DSv4-pro)** — `scripts/bottleneck/aggregate.py`
   groups by cell, computes category/section distributions and
   `Σ (1 - treesim) by section` (failure weight = frequency × severity),
   computes per-task baseline-vs-best deltas, and asks DSv4-pro to
   write a paper-grade synthesis anchored on a chosen baseline/best
   pair.

Total cost across all four task sets: ~650 DSv4-flash calls +
4 DSv4-pro narratives (~$5-7 in DeepSeek API spend).

### Reproducibility

- All cells run on `deepseek-v4-flash` (same model, same parameter
  count); fairness adjustments not needed. Verified that no cell
  uses a different model or different decoding settings.
- `--workers 8`, `--timeout 1500s` per task, `bypassPermissions` mode.
- 3 seeds per cell × task set; reported mean uses
  `with_failures_as_zero_mean` (conservative).
- Source paths: `/data/shared/geophysics_agent_data/data/eval/autocamp_2026-05-01/`
  (Phase 2 + derisk) and `/data/shared/.../autocamp_followup_2026-05-02/`
  (scaleup); scoring under `_results/` and `_results_{icl,train}/`.

## Results

### Quality across distributions

The headline finding is that **adapter value is task-distribution
dependent**:

| set | F0 | F2 | F4 | F6 | F8 | F11 | SE | best Δ vs F0 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| test-17 (n=51) | 0.910 | 0.919 | **0.921** | 0.917 | 0.911 | 0.897 | 0.919 | +1.1pp |
| ICL-10 (n=30)  | 0.720 | — | 0.768 | 0.781 | 0.783 | 0.775 | **0.789** | **+6.9pp** |
| train-19 (n=57)| 0.867 | — | — | 0.869 | — | — | — | +0.2pp |

(F2 ran only on test-17; F4/F8/F11/SE excluded from train-19 by the
contamination rule. F1/F3/F5/F7 — RAG cells — only on test-17 since
they showed -3.3pp main effect there.)

Three patterns:

1. On **test-17**, all reasonable cells lie within 0.91-0.92. The best
   cell (F4) beats baseline (F0) by only +1.1pp, well within seed σ.
2. On **ICL-10**, the same cells span 0.72-0.79; F0 baseline drops to
   0.720 while SE holds at 0.789 (+6.9pp). The plugin's value is
   substantial *and* substantially larger than on test-17.
3. On **train-19**, F0 and F6 are tied at 0.867/0.869 within noise.
   Memory-free adapters add nothing on train-19.

### Reliability

Adapters reduce variance more reliably than they raise mean
(σ values are sample standard deviation across 3 seeds; population
σ are ~18% smaller and what `analyze_autocamp.py` reports):

| set | F0 σ | F4 σ | F6 σ | SE σ |
|---|---:|---:|---:|---:|
| test-17 | 0.024 | 0.007 | 0.004 | 0.020 |
| ICL-10  | 0.081 | 0.005 | **0.002** | 0.012 |
| train-19| 0.012 | – | 0.007 | – |

On ICL-10, F6 reduces seed-to-seed σ by **40×** (0.081 → 0.002). The
adapter regularizes catastrophic seed-2 failures. The mean lift is
real but the reliability lift is more dramatic.

**Best-vs-mean check** (paper-fairness convention: report mean unless
best-mean gap is small). Three cells exceed 3pp gap:
- F0 ICL-10: mean=0.720, best=0.788 (+6.8pp gap, driven by seed 3
  having 1 unparseable XML — counted as 0).
- F11 test-17: mean=0.897, best=0.928 (+3.1pp gap, driven by seed 2
  having 1 unparseable XML).
- F3 test-17: mean=0.857, best=0.890 (+3.3pp gap, similar — F3 is a
  RAG cell, omitted from headline tables but discussed under the
  Resolution-IV main effects).

For these cells we report mean throughout. The pattern (single
seed degenerates → mean drops 3-7pp) is itself diagnostic: F0 and
RAG cells have brittleness that adapter cells (F4, F6, F8, SE)
do not. This is the reliability finding restated.

### Efficiency (tool calls, wall time)

We report mean tool calls and wall time per task; `deepseek-v4-flash`
output tokens are not exposed by the API and so omitted (input tokens
are largely cache reads, ~3-5M per task across all cells).

| cell | tools/task (test-17) | wall s (test-17) | tools/task (ICL-10) | wall s (ICL-10) | reads/task | xmllint/task | RAG/task |
|---|---:|---:|---:|---:|---:|---:|---:|
| F0 | 81.5 | 359 | 90.5 | 417 | 42-45 | 0 | 0 |
| F2 | 72.8 | 349 | – | – | 35 | 0 | 0.5 |
| F4 | 79.6 | 337 | 75.0 | 340 | 35-30 | **2.8** | 2.3 |
| F6 | 83.3 | 348 | 74.7 | 345 | 37-30 | 2.7 | 2.0 |
| F8 | 71.0 | 326 | 82.9 | 358 | 31-34 | 2.7 | 2.6 |
| F11 | 62.7 | 326 | 70.9 | 362 | 25-30 | 2.7 | 2.1 |
| SE | 68.9 | 321 | 97.4 | 390 | 31-43 | 2.7 | 0 |

Three observations:

1. **xmllint cells (F4/F6/SE) make ~3 validation calls per task** —
   the agent self-validates before declaring done. RAG cells (F1/F3/F5/F7,
   omitted from this table) make ~12-13 RAG calls per task.

2. **F0 baseline reads 42-45 files per task** vs F4 at 30-35: the
   memory cheatsheet and xmllint short-circuit some exploratory reads.
   On ICL-10 the gap widens (F0=45 reads, F4=30 reads).

3. **Wall time varies more with task difficulty than cell**: ICL-10
   F0 is 417s vs test-17 F0 359s — the harder tasks need ~16% more
   time. Adapter cells are ~10% faster on test-17 but ICL-10 wall
   times converge (340-390s).

### Where the ICL-10 gain comes from (per-task analysis)

Per-task mean TreeSim on ICL-10 (rows sorted by F0 baseline score):

| task | F0 | F4 | F6 | SE | F0→SE Δ |
|---|---:|---:|---:|---:|---:|
| TutorialHydraulicFractureWithAdvancedXML | 0.013 | 0.013 | 0.013 | 0.013 | 0 |
| AdvancedExampleThermoPoroElasticWellbore | 0.355 | 0.680 | 0.681 | **0.761** | **+0.406** |
| ExampleProppantTest                      | 0.541 | 0.810 | 0.809 | **0.825** | **+0.284** |
| ExampleIsothermalHystInjection           | 0.755 | 0.747 | 0.751 | 0.717 | -0.038 |
| AdvancedExampleCasedThermoElasticWellbore | 0.847 | 0.807 | 0.923 | 0.886 | +0.039 |
| ExamplesingleFracCompression             | 0.891 | 0.887 | 0.904 | 0.928 | +0.037 |
| ExampleVerticalPoroElastoPlasticWellbore | 0.909 | 0.903 | 0.906 | 0.944 | +0.035 |
| ExampleMCCWellbore                       | 0.935 | 0.924 | 0.908 | 0.941 | +0.006 |
| AdvancedExamplePureThermalDiffusionWellbore | 0.963 | 0.922 | 0.956 | 0.880 | -0.083 |
| AdvancedExampleViscoExtendedDruckerPrager | 0.986 | 0.991 | 0.963 | 0.996 | +0.010 |

The **aggregate +6.9pp** F0→SE gain decomposes into:
- 2 catastrophic-failure rescues (ThermoPoroElastic +40.6pp,
  ProppantTest +28.4pp): contribute ~+6.9pp aggregate.
- 1 universal failure (HydraulicFractureWithAdvancedXML, all cells
  ~0.013): a model-level limitation, not a harness one.
- 7 mild changes (±4pp): essentially noise.

In short: **the plugin's job on ICL-10 is to prevent the baseline
from collapsing on 2 hard tasks** (coupled thermal + poro + casing,
and proppant slot fluid mechanics). On easy tasks all cells tie.

### Bottleneck analysis (failure-mode shifts under adapters)

The Stage-2 categorization (DSv4-flash, n=255 on test-17,
n=179 on ICL-10) reveals consistent failure-category distributions:

**test-17 (51 tasks per cell × 7 cells):**

| cell | bad_attr | extra_block | hallucinated | missing | partial | structural |
|---|---:|---:|---:|---:|---:|---:|
| F0 (baseline) | 12 | 9 | 4 | **6** | 7 | 6 |
| F4 (best test-17) | 11 | **11** | **7** | 3 | 6 | 7 |
| SE (v3) | 9 | 10 | – | – | 9 | – |

**ICL-10 (30 tasks per cell × 6 cells):**

| cell | bad_attr | extra_block | hallucinated | missing | partial | structural |
|---|---:|---:|---:|---:|---:|---:|
| F0 (baseline) | 5 | **9** | 0 | 4 | 1 | **8** |
| F4 | 7 | 4 | 0 | 5 | 0 | 6 |
| SE (best ICL-10) | 0 | 5 | 4 | 4 | 5 | 5 |

Three consistent observations across distributions:

1. **`missing_block` is the one category adapters reliably fix.**
   On test-17, F4 reduces missing_block from 6 → 3 (-50%). On ICL-10,
   F4/SE reduce baseline structural_mismatch from 8 → 5-6.
2. **`extra_block` and `hallucinated_extras` get *worse* with adapters**
   on test-17 (9 → 11, 4 → 7). The cheatsheet nudges the agent to
   include patterns it has seen even when the GT does not.
3. **Perfect-task counts do not increase**. F0 has 7 perfect tasks on
   test-17; F4 has 6. Adapters operate in a *harm-reduction* mode,
   reducing the severity of failures without producing more flawless
   outputs.

The DSv4-pro syntheses (XN-019/020/021/022) reach the same conclusion:
schema-aware validation reliably guarantees presence but cannot enforce
content correctness.

## Discussion

### When does an adapter actually help?

The two most informative comparisons in this campaign are
*test-17 vs ICL-10* and *F0 vs SE*. They suggest the following
heuristic:

- **On familiar benchmarks** (test-17, train-19), DSv4-flash reaches
  ~0.92 with no plugin; adapters add < 1pp and are net-zero on
  quality.
- **On out-of-distribution tasks** (ICL-10), the baseline catastrophically
  fails on 1-3 hard tasks while adapters rescue them, yielding +5-7pp
  aggregate gains.

This explains why prior literature on coding-agent harnesses has
produced contradictory results: a benchmark that the model already
"knows" (small distribution gap from training data) shows nearly no
adapter value, while a benchmark that contains genuinely novel task
types makes the same adapter look transformative.

**Implication for benchmark design**: aggregate-mean comparisons on
small in-distribution test sets systematically *understate* the value
of harness adaptations. ICL-10 with 10 tasks identifies the
adapter winner more reliably than test-17 with 17 tasks because the
former has more failure-rescue opportunities.

### What do the adapters actually fix?

The bottleneck analysis disaggregates the aggregate quality
delta into specific structural error classes:

1. **Block presence** (`missing_block`). xmllint+memory reliably catches
   missing top-level sections (e.g., complete omission of `<Events>`,
   `<Solvers>`, `<Constitutive>`). On test-17 this category drops 6→3
   under F4; on train-19 14→8 under F6.
2. **Block content** (`bad_attribute_value`). Schema validation cannot
   enforce *correct* attribute values (only that attributes are typed
   correctly). F4 still has 11 bad_attribute_value failures on test-17
   vs F0's 12 — essentially unchanged. The persistent `gravityVector`
   hallucination on `<Solvers>` (F4 introduces it on multiple ICL-10
   tasks) is a textbook example: the schema permits the attribute, but
   the GT for these specific tasks does not include it.
3. **Block over-generation** (`extra_block`, `hallucinated_extras`).
   These categories *increase* under memory-equipped adapters, not
   decrease. The cheatsheet seeds patterns ("for poromechanics include
   X, Y, Z") that the agent applies more aggressively than the GT
   warrants.
4. **Cross-section consistency**. The train-19 analysis surfaced a
   novel failure class: `<ElementRegion materialList="A,B,C">`
   referencing material names that do not exist in the `<Constitutive>`
   block. Schema validation passes; runtime would fail.

### Concrete adapter-design recommendations

Each recommendation below is grounded in a specific failure category
or section weight that survives the F0 → best-config transition:

1. **Schema-enforced block presence is the cheapest and most reliable
   adapter** — eliminates `missing_block` (~50% reduction) at the cost
   of one xmllint MCP call per task. We recommend this as a baseline
   for any GEOS-XML benchmark harness.

2. **Memory cheatsheets must be paired with negative constraints** —
   our m1u cheatsheet (a list of "for physics X, use solver Y") increases
   `extra_block` and `hallucinated_extras` because the agent applies
   patterns aggressively. Future memory designs should explicitly
   enumerate child-count limits and forbidden elements ("the GT for
   this task family contains exactly 6 Constitutive children, no more").

3. **Attribute-level oracles are the next frontier**.
   `bad_attribute_value` is the single most stable failure category
   (12-15/51 across all cells on test-17, 5-8/30 across all cells on
   ICL-10) and remains untouched by xmllint, memory, or self-evolution.
   An adapter that maintains a per-solver-type allowed-attribute table
   and a per-element allowed-name table would attack this category
   directly. This is a compositionally simple oracle (essentially an
   extended schema with allowed-value constraints), but no current
   open-source GEOS toolchain provides it.

4. **Cross-section consistency hooks** — validate that every
   `<ElementRegion materialList>` entry exists as a `name=` in the
   `<Constitutive>` block. The train-19 analysis surfaced this class
   of failures (mismatched `casing` vs `casingMaterial`); they are not
   caught by schema validation alone and would silently fail at
   simulation time.

5. **Closed-loop retries driven by validator output** —
   the adapter's effect on the *perfect-task count* is essentially nil
   (F0 7/51 → F4 6/51). Static hooks raise the floor on failures but
   do not enable corrections. To produce strictly more correct outputs
   the adapter needs feedback loops: "xmllint says X is missing; add
   X and re-validate."

### Limitations

- **Bottleneck-analysis labels are not adjudicated.** DSv4-flash
  classifies into 8 categories with high consistency but without
  human review. We treat the category counts as descriptive
  rather than ground-truth labels.
- **DSv4-pro syntheses occasionally over-claim mechanism.** The
  ICL-10 narrative (XN-021) misattributes SE's mechanism (claiming
  "post-generation diff against GT" when SE has no GT access). The
  numerical observations are reliable; the *causal* claims need
  manual review.
- **Single model.** All cells use `deepseek-v4-flash`; we have not
  verified that the adapter rankings transfer to other models. Phase 3
  cross-model preflights (minimax-m2.7, gemma-4-31b-it,
  gpt-oss-120b) showed model-capability dominates the picture
  (gemma too slow on OpenRouter; gpt-oss-120b cannot sustain the
  tool-use loop without a Stop hook backstop).
- **TreeSim does not measure simulation correctness.** A plausible
  XML that schema-validates may still produce wrong physics. We have
  not attempted to run GEOS on generated XMLs in this evaluation.
- **train-19 contamination is partial.** m1u was distilled from
  train-18 trajectories, but train-19 (test_36 minus test-17) overlaps
  with train-18 by 18/19 tasks. We restrict train-19 evaluation to
  memory-free cells (F0, F6) to avoid the contamination, but this
  limits the comparisons we can draw on that set.
