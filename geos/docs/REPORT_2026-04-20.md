# repo3 Sprint Report — 2026-04-20

*GEOS XML authoring agent, extended sprint. Plugin characterization, memory experiments, primer ablation, cross-model generalization.*

## Headline

1. **Plugin beats vanilla Claude Code across two models at parity.**
   - deepseek-v3.2: mean TreeSim **0.828 vs 0.653 = +0.178** paired on 35 tasks (29/35 wins, max loss -0.17).
   - minimax-m2.7: mean TreeSim **0.809 vs 0.694 = +0.115** paired on 15 tasks (11/15 wins).
   - Plugin advantage is larger on the weaker base model (deepseek), smaller on the stronger one (minimax). Expected pattern; suggests plugin helps more when base model would otherwise flail.

2. **The plugin's value is systematic, not variance.** Reading raw trajectories on the top-3 deepseek delta tasks (Sneddon +0.705, DPWellbore +0.687, Mandel +0.673) shows the plugin provides **(a) semantic discovery** of alternate solver architectures, **(b) schema-aware filtering** that excludes irrelevant source-code hits, and **(c) variant disambiguation** via documentation context. Grep/Glob/Bash can't reach these because files don't share filename patterns and source-code keywords match irrelevant hits. See XN-008.

3. **Four distinct stacking attempts ALL hurt plugin performance.** They all regress hardest on the same three "rescue tasks" (Sneddon, DPWellbore, Mandel):

   | Attempt | Paired delta (17 test) | Failure mode |
   |---|---:|---|
   | Long cheatsheet in system prompt (E04) | **-0.322** | `redacted_thinking → end_turn`, 3 no-outputs |
   | Short cheatsheet in system prompt (E05) | -0.270 | Same no-outputs pattern + 2 new timeouts |
   | Filetree index in system prompt (E07) | -0.227 | No no-outputs; XML quality degrades |
   | Minimal primer (E08) | -0.235 | 4 timeouts; agent loses orientation |

   **Plugin + full primer + RAG MCP is a tightly-coupled stack. Pieces must stay together.** Future augmentations must add to the tool layer (MCP), not the system-prompt layer.

4. **Six-variant memory battery: ALL NEGATIVE, magnitudes converge on -0.19 to -0.32.**

   | # | Attempt | Delta | Channel | Content |

   All 6 regress hardest on the same rescue tasks (Sneddon, DPWellbore, Mandel). MCP-tool is the least-bad channel: it RESCUES DPWellbore (0.939 vs 0.992) and IMPROVES pknViscosityDominated (+0.142), but still fails Sneddon/Mandel. **The rescue mechanism on those 2-3 tasks is fragile to any perturbation of the plugin+primer stack, regardless of delivery channel or content type.**

5. **CC native memory is a no-op in our pipeline.** Every task's `memory_paths/` directory is empty; zero `memory`-named tool calls across 36 tasks; `/memory` slash command is not even in the agent's command list. Per-task container wipes anyway. Not a confound; nothing to investigate there.

## Methodology decisions (why these choices)

- **Frozen memory at test time.** The user explicitly required that memory not serialize task execution. So all memory designs build the memory before the test run starts; test-time queries are read-only.
- **Train/test split on memory source.** 18/17 split from run4's 35 scored tasks, alternating by score so both subsets cover the difficulty range. Train trajectories source the memory; test tasks evaluate.
- **Delivery-channel variation is the main axis.** After E04/E05 failed identically, we isolated system-prompt-inject as the failure channel and split memory tests into:
  - System-prompt inject (E04, E05) — FAILED
  - Workspace file delivery (E09) — pending
  - MCP-tool-on-demand (E11) — pending
- **Content-type variation as a secondary axis.** Long, short, and structural (filetree) all failed in the system-prompt channel, confirming that channel matters more than content type in our failure mode.
- **Workers=12** confirmed safe on this machine (128 CPU, 900GB RAM, 3 L40S GPUs idle, no OpenRouter 429s observed). Increased from the original 6. No rate-limit pain at 12.
- **1200s timeout** still needed; E03 had 4 successful tasks take >1000s.
- **Skill attribution (I11) not run.** Would require a skill-only vs MCP-only split of the plugin; deprioritized in favor of memory variants.

## What would tighten the paper

- **Multi-seed replication** of E03 and E06. Single-seed claims are defensible as preliminary but reviewers will ask.
- **Opus 4.6 cross-model** for a 3rd model point (SOTA). Cost-deferred.
- **Plugin attribution (I11).** Ship skill-only and MCP-only variants on deepseek; characterize which part drives the gain.
- **Hard-mode eval set.** `mine_examples_v2.py` already generates `required_only_spec` variants. Generating a hard-mode subset and rerunning plugin vs no-plugin on it would show whether adaptations matter more when the agent has less information. If easy mode saturates near 0.83 on plugin+deepseek, adaptations have limited headroom.
- **Memory on a stronger model.** E04–E11 all tested on deepseek. The rescue-fragility may be deepseek-specific; Opus or Claude might handle memory without breaking. One E11-style run on minimax with memory MCP would be informative.
- **"Mid-task" memory trigger.** All tested memory designs fire early (proactively). The fragility may be specifically about EARLY memory consultation. A memory tool the agent calls only when stuck (after N failed attempts) might avoid the early-trajectory perturbation.

## Open questions / not answered yet

- **Why does the primer content matter so much?** E08 removed detailed primer content and plugin rescue broke on the same 3 tasks that memory disturbs. Is this about schema navigation, documentation map, or something else? A more surgical primer ablation (e.g., primer-with-RAG-instructions vs primer-with-only-scaffolding) would help.
- **Is the `redacted_thinking → end_turn` failure mode specific to deepseek via OpenRouter/SiliconFlow?** Not tested on minimax or Opus. A single swap-model experiment on a failing task with cheatsheet would tell us quickly.
- **Does the plugin's value saturate at stronger models?** Minimax delta (+0.115) is notably smaller than deepseek (+0.178); extrapolating to Opus, plugin may be marginal. This affects paper framing.
- **The fragile rescue mechanism:** all 6 augmentation attempts regress worst on TutorialSneddon, ExampleMandel, ExampleDPWellbore. Why are these 3 tasks especially sensitive to any perturbation? The trajectory analysis (XN-008) showed these rely on semantic discovery of non-obvious reference XMLs. Any change in system prompt or tool set seems to interfere with that specific discovery chain.
